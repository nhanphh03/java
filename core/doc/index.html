Chủ đề 01
Java Stream API
Xử lý dữ liệu theo phong cách lập trình hàm — declarative, lazy, parallel
Stream là gì?

Stream là một luồng xử lý dữ liệu theo từng phần tử, hỗ trợ các phép biến đổi (transformation) và tổng hợp (reduction) theo phong cách functional programming. Stream không phải là cấu trúc dữ liệu — nó không lưu trữ dữ liệu mà chỉ xử lý.

💤
Lazy Evaluation
Các intermediate operation chỉ thực thi khi terminal operation được gọi. Không lãng phí tài nguyên.
🔗
Pipeline
Kết nối chuỗi operations lại với nhau: source → filter → map → collect. Dữ liệu chảy qua pipeline.
🚀
Parallel Stream
Tự động chia nhỏ dữ liệu và xử lý song song trên nhiều CPU core qua ForkJoinPool.
🔒
Immutable
Không thay đổi source data. Mỗi operation tạo ra một stream mới chứ không sửa stream gốc.

Parallel Stream — Xử lý song song
⚠️

Parallel Stream không phải lúc nào cũng nhanh hơn! Chỉ hiệu quả khi: dữ liệu lớn (hàng nghìn phần tử+), tác vụ CPU-intensive, không có shared mutable state, collection hỗ trợ spliterator tốt (ArrayList, array).

ParallelStream.javaJava
// Parallel stream: thêm .parallel() hoặc dùng .parallelStream()
long sum = LongStream.rangeClosed(1, 1_000_000)
    .parallel()
    .sum();  // Tự chia tác vụ cho các CPU core

// ❌ Sai: dùng shared mutable state trong parallel stream
List<Integer> badResult = new ArrayList<>();
nums.parallelStream()
    .forEach(badResult::add);  // ⚠️ Race condition!

// ✅ Đúng: dùng collect() thread-safe
List<Integer> goodResult = nums.parallelStream()
    .filter(n -> n % 2 == 0)
    .collect(Collectors.toList());  // ✅ An toàn

Chủ đề 02
Generics nâng cao & Wildcards
Type safety tại compile-time, wildcards, bounded types và type erasure
Tại sao cần Generics?

Generics cho phép định nghĩa class, interface, method với kiểu dữ liệu là tham số. Thay vì viết code riêng cho từng kiểu, bạn viết một lần và Java đảm bảo type safety tại compile-time.

WhyGenerics.javaJava
// ❌ Không có generics — mất type safety
List rawList = new ArrayList();
rawList.add("hello");
rawList.add(42);       // OK lúc compile, nhưng...
String s = (String) rawList.get(1); // 💥 ClassCastException lúc runtime!

// ✅ Có generics — lỗi bị bắt ngay khi compile
List<String> typedList = new ArrayList<>();
typedList.add("hello");
// typedList.add(42);  ← Compile error: incompatible types
String val = typedList.get(0); // Không cần cast

// Generic class: Pair<A, B>
public class Pair<A, B> {
    private final A first;
    private final B second;

    public Pair(A first, B second) {
        this.first = first;
        this.second = second;
    }

    public A getFirst()  { return first; }
    public B getSecond() { return second; }
}

Pair<String, Integer> p = new Pair<>("age", 25);
String key = p.getFirst();   // "age"
int    age = p.getSecond();  // 25
Bounded Type Parameters — Giới hạn kiểu
BoundedTypes.javaJava
// Upper Bound: T extends Comparable — T phải implement Comparable
public static <T extends Comparable<T>> T findMax(List<T> list) {
    return list.stream()
               .max(Comparator.naturalOrder())
               .orElseThrow();
}

findMax(List.of(3, 1, 7, 2));  // 7
findMax(List.of("B", "A", "C")); // "C"

// Multiple bounds: T phải là Animal VÀ implement Serializable
public <T extends Animal & Serializable> void save(T animal) { ... }

// Generic method — tính tổng list số
public static <N extends Number> double sumList(List<N> list) {
    return list.stream()
               .mapToDouble(Number::doubleValue)
               .sum();
}

sumList(List.of(1, 2, 3));       // Integer — OK
sumList(List.of(1.5, 2.5));     // Double  — OK
Wildcards — ? và PECS

Wildcard ? đại diện cho "kiểu không xác định". Có 3 dạng chính:

? (Unbounded)
List<?> — Có thể là bất kỳ kiểu nào. Chỉ đọc được (get), không thêm được (add).
? extends T (Upper)
List<? extends Number> — Có thể là Number hoặc subtype. Dùng để đọc (Producer).
? super T (Lower)
List<? super Integer> — Có thể là Integer hoặc supertype. Dùng để ghi (Consumer).
💡

PECS Rule — Producer Extends, Consumer Super: Nếu collection cung cấp dữ liệu ra ngoài → dùng extends. Nếu collection nhận dữ liệu vào → dùng super.

PECS_Wildcards.javaJava
// PRODUCER EXTENDS — đọc từ source để tính tổng
// source "sản xuất" Number để ta đọc → extends
public static double sumProducer(List<? extends Number> source) {
    double total = 0;
    for (Number n : source) total += n.doubleValue();
    return total;
}

sumProducer(List.of(1, 2, 3));    // List<Integer> — OK
sumProducer(List.of(1.1, 2.2));  // List<Double>  — OK

// CONSUMER SUPER — ghi vào dest
// dest "tiêu thụ" Integer ta đưa vào → super
public static void addNumbers(List<? super Integer> dest) {
    dest.add(1);
    dest.add(2);
    dest.add(3);
}

List<Integer> intList    = new ArrayList<>();
List<Number>  numList    = new ArrayList<>();
List<Object>  objList    = new ArrayList<>();
addNumbers(intList);  // OK
addNumbers(numList);  // OK
addNumbers(objList);  // OK

// Ví dụ thực tế — copy từ src sang dest (giống Collections.copy)
public static <T> void copy(
    List<? extends T> src,    // Producer (đọc)
    List<? super T> dest      // Consumer (ghi)
) {
    for (T item : src) dest.add(item);
}
Type Erasure — Thực tế ở runtime
ℹ️

Java sử dụng Type Erasure: thông tin generic bị xóa hoàn toàn sau compile-time. Tại runtime, List<String> và List<Integer> đều là List. Điều này giải thích tại sao không thể dùng instanceof List<String> hay tạo new T[].

TypeErasure.javaJava
// Compile time → Runtime (sau erasure)
List<String> strings;   →  List strings;
List<Integer> ints;    →  List ints;

// Hệ quả: không thể phân biệt tại runtime
List<String> ls = new ArrayList<>();
List<Integer> li = new ArrayList<>();
System.out.println(ls.getClass() == li.getClass()); // true!

// ❌ Không được:
if (obj instanceof List<String>) {} // Compile error
T[] arr = new T[10];               // Compile error

// ✅ Workaround: truyền Class<T> vào
public <T> T deserialize(String json, Class<T> type) {
    return objectMapper.readValue(json, type);
}
User user = deserialize(jsonStr, User.class);

💳

⚡

LoggingAspect.java (Spring AOP)Java
@Aspect
@Component
public class LoggingAspect {

    // Pointcut: tất cả method trong package service
    @Pointcut("execution(* com.app.service.*.*(..))")
    public void serviceLayer() {}

    // ① @Before — chạy TRƯỚC method
    @Before("serviceLayer()")
    public void logBefore(JoinPoint jp) {
        System.out.println("▶ Calling: " + jp.getSignature().getName());
    }

    // ② @AfterReturning — chạy SAU khi return bình thường
    @AfterReturning(pointcut = "serviceLayer()", returning = "result")
    public void logAfterReturn(JoinPoint jp, Object result) {
        System.out.println("✅ Return: " + result);
    }

    // ③ @AfterThrowing — chạy KHI ném exception
    @AfterThrowing(pointcut = "serviceLayer()", throwing = "ex")
    public void logException(JoinPoint jp, Exception ex) {
        System.out.println("❌ Exception in: " + jp.getSignature() + " — " + ex.getMessage());
    }

    // ④ @After — chạy SAU BẤT KỂ (finally)
    @After("serviceLayer()")
    public void logFinally(JoinPoint jp) {
        System.out.println("⏹ Done: " + jp.getSignature().getName());
    }

    // ⑤ @Around — mạnh nhất, bao toàn bộ method
    @Around("serviceLayer()")
    public Object measureTime(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            Object result = pjp.proceed();  // gọi method thực sự
            return result;
        } finally {
            long elapsed = System.currentTimeMillis() - start;
            System.out.println("⏱ " + pjp.getSignature() + " took " + elapsed + "ms");
        }
    }
}
Ví dụ thực tế: Custom Annotation + AOP
RateLimiter_AOP.javaJava
// ① Định nghĩa custom annotation
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface RateLimit {
    int maxRequests() default 100;
    int perSeconds()  default 60;
}

// ② Dùng annotation trên method
@Service
public class PaymentService {

    @RateLimit(maxRequests = 10, perSeconds = 60)
    public Receipt processPayment(Payment payment) {
        // chỉ business logic, không có rate-limit code ở đây
        return paymentGateway.charge(payment);
    }
}

// ③ Aspect xử lý @RateLimit tự động
@Aspect @Component
public class RateLimitAspect {

    private final Map<String, RateLimiter> limiters = new ConcurrentHashMap<>();

    @Around("@annotation(rateLimit)")
    public Object enforceRateLimit(ProceedingJoinPoint pjp, RateLimit rateLimit)
        throws Throwable {

        String key = pjp.getSignature().toLongString();
        RateLimiter limiter = limiters.computeIfAbsent(key,
            k -> RateLimiter.create(rateLimit.maxRequests() / (double) rateLimit.perSeconds()));

        if (!limiter.tryAcquire()) {
            throw new TooManyRequestsException("Rate limit exceeded");
        }
        return pjp.proceed();
    }
}

Java Reflection API
Kiểm tra và thao tác cấu trúc class tại runtime — meta-programming trong Java
Reflection là gì?

Reflection cho phép chương trình kiểm tra và thao tác cấu trúc của chính nó tại runtime: đọc class name, list fields/methods, gọi method theo tên, tạo object mà không biết class tại compile time. Đây là nền tảng của Spring, JPA, Jackson, JUnit.

⚠️

Reflection có chi phí hiệu năng cao hơn gọi trực tiếp và bypass các kiểm tra compile-time. Chỉ dùng khi thực sự cần (framework, serialization, DI container).

Class Object — Cửa vào Reflection

System.out.println(clazz.getSimpleName());    // User

// Lấy Methods

Ứng dụng thực tế: Mini Dependency Injection
MiniDI.javaJava
// Custom @Inject annotation
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface Inject {}

// DI Container đơn giản
public class Container {
    private Map<Class<?>, Object> beans = new HashMap<>();

    public <T> T get(Class<T> type) throws Exception {
        if (beans.containsKey(type)) return type.cast(beans.get(type));

        T instance = type.getDeclaredConstructor().newInstance();

        // Inject tất cả field có @Inject
        for (Field f : type.getDeclaredFields()) {
            if (f.isAnnotationPresent(Inject.class)) {
                f.setAccessible(true);
                f.set(instance, get(f.getType())); // đệ quy resolve
            }
        }
        beans.put(type, instance);
        return instance;
    }
}

// Sử dụng
public class OrderController {
    @Inject OrderService orderService;  // tự động inject
    @Inject UserService  userService;
}

Container container = new Container();
OrderController ctrl = container.get(OrderController.class);
// orderService và userService đã được inject tự động!

Chủ đề 05
CompletableFuture
Lập trình bất đồng bộ hiện đại — composition, chaining, error handling
Tại sao CompletableFuture?

Future cũ (Java 5) chỉ cho phép chờ kết quả bằng get() — block thread. CompletableFuture (Java 8) cho phép compose, chain, combine nhiều tác vụ async mà không block thread, với API phong phú và error handling linh hoạt.

Tạo CompletableFuture
CF_Creation.javaJava
// runAsync — không trả về giá trị (Runnable)
CompletableFuture<Void> f1 = CompletableFuture.runAsync(() -> {
    System.out.println("Running in: " + Thread.currentThread().getName());
});

// supplyAsync — có trả về giá trị (Supplier)
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> {
    // giả lập gọi API mất 1 giây
    sleep(1000);
    return "User data from API";
});

// Dùng custom Executor thay vì ForkJoinPool.commonPool()
ExecutorService pool = Executors.newFixedThreadPool(10);
CompletableFuture<String> f3 = CompletableFuture.supplyAsync(
    () -> fetchFromDB(), pool
);

// completedFuture — wrap giá trị có sẵn (test, fallback)
CompletableFuture<String> f4 = CompletableFuture.completedFuture("cached value");

Chủ đề 06
Optional Class
Loại bỏ NullPointerException — xử lý giá trị có thể null một cách tường minh
Optional là gì và tại sao dùng?

Optional<T> là container có thể chứa hoặc không chứa giá trị non-null. Thay vì trả về null (dẫn đến NPE), method trả về Optional — buộc caller phải xử lý trường hợp không có giá trị.

Optional_Creation.javaJava
// Tạo Optional
Optional<String> empty   = Optional.empty();           // rỗng
Optional<String> present = Optional.of("hello");       // non-null
Optional<String> maybe   = Optional.ofNullable(name);  // null OK

// ❌ Anti-pattern: trả về null
public User findUser(int id) {
    return db.find(id); // có thể null → NPE ở caller
}

// ✅ Đúng: trả về Optional
public Optional<User> findUser(int id) {
    return Optional.ofNullable(db.find(id));
}
Các method quan trọng
Optional_Methods.javaJava
Optional<User> opt = findUser(42);

// Kiểm tra
opt.isPresent();   // true nếu có giá trị
opt.isEmpty();     // true nếu rỗng (Java 11+)

// Lấy giá trị
opt.get();                            // ⚠️ throw nếu empty
opt.orElse(User.guest());            // default value
opt.orElseGet(() -> createGuest()); // lazy default (Supplier)
opt.orElseThrow(() -> new NotFoundException("User not found"));

// Transform — tất cả trả về Optional (không throw nếu empty)
opt.map(User::getName);             // Optional<String>
opt.flatMap(User::getAddress);     // khi getAddress trả Optional
opt.filter(u -> u.isActive());     // Optional<User> hoặc empty

// Hành động
opt.ifPresent(u -> sendEmail(u));  // chạy nếu có giá trị
opt.ifPresentOrElse(                  // Java 9+
    u -> sendEmail(u),
    ()  -> System.out.println("User not found")
);

// or() — Java 9+: fallback sang Optional khác
Optional<User> result = findInDB(id)
    .or(() -> findInCache(id))
    .or(() -> findInBackup(id));
Chaining Optional — Tránh NPE trong chuỗi dài
Optional_Chaining.javaJava
// ❌ Cũ: kiểm tra null thủ công từng bước
String city = null;
User user = getUser(id);
if (user != null) {
    Address addr = user.getAddress();
    if (addr != null) {
        city = addr.getCity();
    }
}

// ✅ Mới: Optional chain
String city = findUser(id)
    .flatMap(User::getOptionalAddress)  // Optional<Address>
    .map(Address::getCity)              // Optional<String>
    .orElse("Unknown City");            // String

// Ví dụ phức tạp: tìm giảm giá của user
double discount = findUser(userId)
    .filter(User::isPremium)
    .flatMap(User::getMembership)
    .map(Membership::getDiscountRate)
    .orElse(0.0);

// ⚠️ KHÔNG nên làm: Optional cho field, collection item
// Optional chỉ dùng cho return type của method, không phải field
class BadExample {
    private Optional<String> name; // ❌ Không làm thế này
}
💡

Best practices: Dùng Optional làm return type — không bao giờ làm parameter, field, hay trong collection. orElse() luôn evaluate expression dù có giá trị; dùng orElseGet() (lazy) khi expression tốn chi phí.

Memory Leaks — Nguyên nhân và phòng tránh
MemoryLeaks.javaJava
// ❌ Leak 1: Static collection tích lũy không xóa
static Map<String, Session> sessions = new HashMap<>();
// sessions chưa bao giờ xóa entry cũ → leak
// ✅ Fix: sessions.remove(key) khi logout, hoặc dùng WeakHashMap

// ❌ Leak 2: Listener không unregister
public class MyComponent {
    public void init() {
        eventBus.subscribe("event", this::handle);
    }
    // ❌ chưa unsubscribe → GC không thể collect MyComponent
    // ✅ Fix: unsubscribe trong destroy()/close()
    public void destroy() {
        eventBus.unsubscribe("event", this::handle);
    }
}

// ❌ Leak 3: Inner class giữ reference đến outer class
class Outer {
    private byte[] data = new byte[1024 * 1024];  // 1MB
    class Inner { ... }  // Inner giữ tham chiếu đến Outer!
    // ✅ Fix: dùng static nested class
    static class StaticInner { ... }
}

// ❌ Leak 4: ThreadLocal không cleanup
static ThreadLocal<Session> threadLocal = new ThreadLocal<>();
public void handleRequest() {
    threadLocal.set(new Session());
    try {
        // xử lý
    } finally {
        threadLocal.remove(); // ✅ PHẢI remove() sau khi dùng xong
    }
}
